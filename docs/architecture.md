# 系统架构设计

## 1. 总体目标

本系统拟实现一个以 VS Code 插件为载体的编程助手 Agent，强调以下三个维度的融合：

- 开发环境原生集成
- 智能体式任务协作
- 虚拟角色化交互体验

系统面向毕业设计场景，既要具备可展示的原型能力，也要具备继续扩展为实验平台的结构基础。

## 2. 分层架构

### 2.1 表现层

表现层由 VS Code Webview 构成，负责：

- 聊天式任务输入
- 代码建议展示
- 当前上下文状态提示
- 虚拟角色区域展示

### 2.2 Agent 编排层

Agent 层负责：

- 接收用户请求
- 读取编辑器上下文
- 构造系统提示词与任务提示
- 调用模型适配器
- 输出结构化响应

后续可扩展工具链：

- 工作区文件检索
- 代码差异生成
- 终端命令建议
- 测试生成与执行

### 2.3 模型适配层

模型适配层以统一接口封装不同推理来源：

- Mock Provider：用于演示与界面联调
- Local Provider：接入本地推理服务
- Remote Provider：后续可扩展到云端模型

LoRA 微调后的模型建议通过本地服务统一暴露 HTTP API，插件只负责请求与结果渲染，不直接承担训练逻辑。

### 2.4 角色驱动层

角色驱动层负责虚拟人物的状态映射：

- 待机
- 思考
- 回答
- 鼓励反馈

在原型阶段先使用状态机与占位 UI 表示，后续接入 VRM 运行时后，可将 Agent 输出映射到：

- 表情
- 头部动作
- 身体姿态
- 口型同步

## 3. LoRA 微调方案建议

### 3.1 目标

通过 LoRA 对基础大模型进行轻量化微调，使模型更适应编程助手场景，重点增强：

- 中文技术问答风格一致性
- 代码解释与注释生成能力
- 插件命令式交互格式
- 教学型、陪伴型回复风格

### 3.2 数据来源建议

- 开源代码问答数据
- 自建代码解释样本
- 论文场景下整理的“编程助手对话数据”
- 以 VS Code 使用场景为背景构造的任务数据

### 3.3 部署建议

- 微调平台：QLoRA / LoRA
- 推理框架：vLLM、Ollama、LM Studio 或自建 FastAPI 服务
- 插件对接方式：HTTP JSON API

## 4. 毕设展示建议

演示时建议展示以下流程：

1. 用户在 VS Code 中输入需求
2. Agent 读取当前文件上下文
3. 模型返回代码建议或解释
4. 虚拟角色同步切换状态
5. 展示本地 LoRA 模型与通用模型在回复风格或任务表现上的差异
